# Accessibility and Inclusive UX Checklist

## Documentation Accessibility

- Heading hierarchy is logical and complete
- Link text is descriptive and unambiguous
- Instructions avoid jargon without definition
- Long code examples include context before and after
- Key warnings are visible without relying on color only

## Interaction Accessibility (Swagger + API Flow)

- Keyboard-only navigation works for core actions
- Focus order is predictable
- Error messages are specific and actionable
- Required fields are clearly labeled
- Example payloads are easy to locate and copy

## Inclusive Language

- Avoid culture-specific idioms in core instructions
- Define ambiguous terms (`source`, `storage`, `index`) consistently
- Prefer short action-first steps
- Avoid assumptions about prior API experience

## Cognitive Load Checks

- First-run flow is visible as a sequence, not scattered references
- Fallback and troubleshooting paths are explicitly linked
- Critical branch decisions are called out where they occur

## Reporting Format

For each issue, record:
- Severity
- Affected users
- Evidence
- Proposed wording/layout fix
- Validation method
